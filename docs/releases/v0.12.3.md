# Lyra v0.12.3

**Hardware / scope:** Hermes Lite 2 / 2+ over HPSDR Protocol 1. RX, plus
CW / SSB / AM / DSB / SAM / FM and digital (TCI / VAC) transmit. **Windows
only** for now (Linux / macOS planned, not yet buildable).

A quick fix release: the space-bar PTT change plus two TCI robustness fixes.
(Supersedes the short-lived v0.12.2, which had only the PTT change.)

## Space-bar PTT is now a toggle

Some of you reported the space-bar PTT feeling unstable — occasionally
getting stuck transmitting. Fixed by changing how it works:

- **Press space once → transmit on. Press space again → transmit off.**

This matches Thetis and a normal transceiver's PTT-lock, so nothing new to
learn. It also removes the cause of the stuck-transmit: the old "hold to
talk" behaviour needed the app to reliably see the key *release*, and if
Lyra lost focus mid-transmission (Alt-Tab, a pop-up, clicking another
window) that release could go missing and leave the radio keyed. A toggle
doesn't depend on the release at all. Holding space does nothing odd, and
un-keying always works. (On by default; turn it off in **Settings →
Hardware** next to the hardware-PTT option.)

## TCI: better default port + visible errors

If your TCI server occasionally refused to start — the "TCI server running"
tick box wouldn't stay checked — this is for you.

- **The default TCI port is now 40001** (was 50001). 40001 is the standard
  TCI port *and* it sits outside the Windows temporary-port range, so it
  can't be randomly grabbed by another program at boot. (50001 is inside
  that range — that's how an unrelated app, e.g. a backup service, could
  occasionally snatch it before Lyra and block TCI. It looked like a Lyra
  bug but was a port collision.) **Your existing port setting is kept** — if
  you were on 50001 and hit this, just change the TCI **Port** in
  **Settings → Network** to 40001 (and point your logger/digital app at the
  same port).
- **A failed start is now visible.** Instead of the tick box silently
  bouncing back, Settings → Network shows a red *"Couldn't start — port N in
  use — … (try another port)"*, and the reason is written to the diagnostic
  log. No more guessing.

carries over.
